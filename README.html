数据 现状 问题 备注
用户 保存在 IAM 和 IDaaS * 一致性保证：IAM 写用户数据到 MySQL 和 IDaaS
* 双向同步：IDaaS 的通讯录未同步到 IAM
用户组 * IAM 有用户组数据
* IDaaS 有用户组数据		 * 重复定义：IAM 和 IDaaS 各自独立
* 同步中断：无法将 IDaaS 的用户组数据同步到 IAM
组-用户 关系 * 保存在 IAM 和 Auth
* BFF 同步读/写 IAM 和 Auth 的关系数据		 * 数据冗余：IAM 和 Auth 各存一份关系数据
* 一致性保障：IAM 和 Auth 的关系数据没法确保一致		 IDaaS: 一个用户可以归属于多个用户组
其他 * COS: 用来存储 UI 个性化配置，目前主要是 EnerLink 使用
* Kafka: 消息通知，如发送短信、邮件等，目前主要是 EnerLink 使用		 业务依赖与技术依赖不匹配：Kafka 是服务启动的强依赖

权限

数据 现状 问题 原因
权限项 保存在 BFF 的配置文件 permissions.json 中 * 数据低内聚：权限项与其他信息离散
* 高耦合：BFF 与 Auth 强耦合		 * 担心影响 auth 的通用性
* 动态权限：bff 启动时去拉
角色 保存在 Auth 中 * 未与 IDaaS 同步
角色-用户 关系 保存在 Auth 中 * 未与 IDaaS 同步
权限项-角色 关系 保存在 Auth 中 * 未与客户权限系统同步

统一功能

单点登录

sequenceDiagram actor u as User participant w as Web participant b as BFF-Svr participant i as IAM-Svr participant id as IDaaS u->>id: 登录 id->>id: 校验账户信息，登录成功 id-->>u: 重定向到 Twin u->>w: 访问 w-->>u: 响应
现状 问题
使用 OAuth2 协议，借助 IDaaS 做 SSO * 因通讯录只是单向同步 (IAM -> IDaaS)，导致 IDaaS 新增的用户无法登录 IAM

授权

sequenceDiagram actor u as User participant w as Web participant b as BFF-Svr participant a as Auth-Svr u->>w: 访问 w->>b: HTTP note right of b: BFF 参与授权逻辑 b->>b: 读取权限项 b->>a: 授权 note right of a: Auth 参与授权逻辑 a->>a: 鉴权 a-->>b: 响应 b-->>w: 响应 w-->>u: 响应
现状 问题
授权逻辑由 BFF 和 Auth 协作完成 高耦合
因基于 Casbin，Auth 的每个服务节点均加载全量权限数据到进程
（当权限变更时，通过 Redis pub/sub 进行通知）		 高成本：总内存占用与实例数成正比翻番

三. 方案

问题汇总

认证

问题 解决思路 遗留问题/风险
数据一致性：EnerTwin 用户同步到 IDaaS 时可能中途失败 异步同步：同步写本地 MySQL，异步任务同步数据到 IDaaS 最终一致性：数据延迟较高
单向同步：IDaaS 的新增用户无法登录 EnerTwin 双向同步：将 IDaaS 的新增用户同步到 IAM
用户组数据无法同步：未打通 EnerTwin 和 IDaaS 的用户组数据，导致客户系统的用户组数据无法同步到 EnerTwin 对齐并双向同步用户组 1. IDaaS: 一个用户可以归属于多个用户组？Y
2. 是否需要支持“部门”的同步？
数据冗余：组-用户 关系在 IAM 和 Auth 各存一份数据，且存在不一致风险 只存一份关系数据在 IAM 中
业务依赖与技术依赖不匹配：Kafka 是服务启动的强依赖 缺省时逻辑：技术依赖与业务对齐
小结

IAM 中存储的以下数据并与 IDaaS 双向异步 同步：

  1. 用户
  2. 用户组
  3. 组-用户 关系

授权

问题 解决思路 遗留问题/风险
数据低内聚：权限项离散在 BFF 中 权限项存放到 Auth 中 1. 动态权限如何同步？Auth 中通过 cron 定时同步并持久化
高耦合：BFF 与 Auth 强耦合 授权逻辑由 Auth 独立完成：
1. IAM: 用“用户”换“角色”
2. Auth：对角色（实体） 进行鉴权
注意： 此时的 Auth 已经不是 RBAC 模型，而是简单的 ACL 模型 （此时 Auth 只关心有/没有权限 bool: true/false，通用性可能较好）		 1. 单用户如何授权？单用户独占一角色（Auth 目前如此）
角色数据未与 IDaaS 同步 角色等于用户组，由 IAM 管理（并与 IDaaS 同步），Auth 不存不管
角色-用户 关系数据未与 IDaaS 同步 该关系等于 用户组-用户 关系，由 IAM 管理（并于 IDaaS 同步），Auth 不存不管
权限项-角色 关系未与客户系统同步 多种思路：
1. 客户授权系统 wrapper：实时代理，将授权请求转换逻辑和转发到客户授权系统 API
2. 同步数据：cron 定时同步和转换关系数据		 1. 耗时：可能多次调用客户 API 才能完成授权，网络请求高耗时
2. 授权模型差异：数据是否可转换成功？
高成本：总内存占用高 外部共享存储：权限数据存放到 Redis 中 耗时会增加 1-7ms
小结

Auth 中存放以下数据：

  1. 权限项
  2. 权限项-实体（角色） 关系

注意：

  1. 这里不需要存“用户”和“实体（角色）”的详情数据。
  2. 实体：请求权限的实体的抽象，不管是用户、用户组、角色、App 等都可以

架构图

设计原则

  1. 适配层：屏蔽外部服务差异，让 EnerTwin 服务自治
  2. 服务通用性：IAM 和 Auth 不与特定业务耦合，可在 Link 和 Twin 等项目中使用
    1. IAM: 只管用户、用户组、关系数据和单点登录，与业务逻辑无关
    2. Auth: 指关心有/没有权限，非特定的授权模型
